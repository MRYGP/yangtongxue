常见坑清单

- 指令里出现括号类符号，模型输出自检会回环
- Attempt Log 行数上限太紧，STOP 一加就超
- Source 出现多次或出现在非验收块，触发自检重写
- API 可读但 base64 解码失败，要直接用同一 JSON 的 download_url
- 同一 URL 失败后换理由重试，会浪费打开次数
- INDEX 写成 URL 链接而不是相对路径，会违反 URL 输出禁令
- 日志里写了可识别信息，后期很难清洗
- 占位符不要用任何括号或尖括号，统一用大写词做占位符，比如 PATH、PROJECT_SLUG、YYYY_WWW
- Attempt Log 行数上限要容纳 STOP，建议上限设为 7 行，STOP 算 1 行
- Source 只允许出现在验收呈现块，同步摘要里不写 Source，避免同一文件出现两次 Source
- 输出禁令要覆盖全角符号，除了半角括号方括号，也要禁止全角括号与全角方括号
- 同一 URL 失败后不得用不同理由重复打开，失败即记一次预算，直接走兜底或降级粘贴



## 1) 可复制的协作模式：ChatGPT 负责架构与验收，GPTS 负责契约与读取，Cursor 负责落盘修改

### 角色分工

- 你：项目 Owner，拍板优先级与资源，决定公开信息边界
- ChatGPT：架构师与红队，负责把需求变成“可执行契约”，并给出可落盘的文稿与修改指令
- GPTS：仓库读取与格式验收器，按 v2.3.1 健康检查模式做 3-step 读取、字段提取、周一致性、Attempt Log
- Cursor：唯一写入者，负责在 GitHub 仓库里改文件、加文件、提交与推送

### 标准工作流：一轮闭环怎么跑

1. 定义本轮目标
    - 例：把健康检查跑通、把 JD 写成可投放版本、把 SSOT 模板补齐
2. 写“契约”而不是写“作文”
    - 例：SYNC 必须包含哪些字段、Plan 必须有哪些字段、输出禁止哪些符号
3. 先跑健康检查再做业务推进
    - 目的是确认：读取路径、sha、download_url 兜底、输出自检都能稳定工作
4. 由 ChatGPT 产出两类东西
    - Cursor 修改指令：具体改哪里、替换什么、插入什么
    - 可直接入库的 Markdown 正文：新文件内容一次性给全
5. Cursor 落盘
    - 按指令修改，提交信息写清楚：做了什么、为什么
6. GPTS 再跑一次健康检查
    - 验证：字段提取与一致性 OK，Attempt Log 不越界，URL 白名单不误触发
7. 记录到日志
    - 日志只记录“做了什么、证据是什么、下一步是什么”，不写大段感想

### 最容易踩的坑清单：你这套系统以后复制时最值钱的避坑点

- 坑 1：把“测试跑通”当成“正常推进”
     解决：健康检查触发器必须优先级最高，测试时只做读取与校验，不做计划建议
- 坑 2：输出格式污染
     典型：Markdown 超链接、括号方括号、脚注引用、思考过程泄漏
     解决：输出格式自检写死，不通过就整段重写
- 坑 3：打开次数无限重试
     解决：每文件 2 次打开硬上限，超过就 STOP 改 user_paste
- 坑 4：API content 解码失败还死磕 base64
     解决：API 只要拿到 sha 且有 download_url，就用 download_url 读正文，仍算 API-first
- 坑 5：搜索结果把你带去奇怪网站
     解决：域名白名单，只允许 api.github.com、raw.githubusercontent.com、github.com，其它一律忽略
- 坑 6：JD 里泄露过多个人信息
     解决：可以写雨花区与年级年龄，但不写学校班级、精确小区门牌、孩子姓名与照片
- 坑 7：占位符写成 <xxx> 或类似括号形态
     解决：一律改为大写占位符词，比如 PATH、PROJECT_SLUG，不用任何括号类符号
- 坑 8：Source 在不同结构块重复出现
     解决：Source 只出现在验收呈现块，同步摘要只写状态与关键字段是否齐全

六、可复制工作流 SOP 以后每个项目直接照抄

1 需求澄清
把目标写成一句话成功标准写成三条可验证

2 指令补丁
只做最小改动 不重写全文
每次补丁必须能落到一条失败案例上

3 回归测试 三条就够
测试一 健康检查模式 能读到 SYNC Plan Retro
测试二 输出中不出现被禁止的格式
测试三 Source 每份文件只出现一次 Attempt Log 行数不超上限

4 Cursor 改仓库
一次提交只做一个主题
提交信息模板
feat: xxx
fix: xxx
docs: xxx

5 冻结版本
通过回归测试后 在指令顶部写版本号与变更点
并在 daily log 记录本次通过的测试用例

七、常见坑补充
同一 URL 失败不要用不同理由反复重试
Source 不要在同步摘要里重复出现
占位符不要用尖括号 一律用大写词
